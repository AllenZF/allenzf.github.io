追踪了什么？

硬件辅助追踪（Hardware-Assisted Trace）是一种利用处理器内部集成的专用硬件模块，对程序执行过程进行实时、细粒度记录的技术。与纯软件实现的追踪（如插桩或日志记录）相比，硬件辅助追踪具有低开销、高精度、实时性强等优势。

指令追踪是一种收集处理器执行历史及其他事件的系统。该追踪系统可通过基于寄存器的接口进行设置和控制。hart 的执行活动出现在输入端口，并输入到追踪编码器中，其中数据被压缩并格式化为追踪消息。追踪编码器将追踪消息传输至追踪汇流排。在多核系统中，每个 hart 拥有独立的追踪编码器，通常所有编码器都会连接到一个追踪汇流排，该汇流排汇聚来自多个源的追踪数据并将其发送至单一目的地。

RV 的数据追踪主要追踪4种访问：load, store, amo, csr.

基本原理：

  • 程序执行流捕获：通过跟踪程序计数器（PC）的变化，记录指令的执行路径。
  • 数据访问追踪：监控对内存或寄存器的数据读写操作。
  • 事件触发机制：配合触发器（trigger）与断点（breakpoint）模块，在特定条件（如访问某地址、执行某指令）下启动或停止追踪。
  • 压缩与缓冲：为减少带宽和存储开销，硬件通常支持对追踪数据进行压缩，并通过片上缓冲区（如 trace buffer）暂存数据，再通过调试接口（如 JTAG、ETM 或自定义 trace port）输出。

RV 标准追踪结构



Trace Message/Packet

跟踪消息/数据包 - 根据协议不同，可能使用不同的名称，但含义相同。它被视为连续的（通常为字节）序列，描述程序和/或数据流及其他事件。

  • Branch Trace Messaging
    分支跟踪消息是最简单、基础形式的指令跟踪。每个程序计数器的不连续性都会产生一条跟踪消息，即直接或间接分支消息。线性指令（或线性指令序列）不会直接生成任何跟踪消息/数据包，但计数器溢出（或异常）可能会生成相应的数据包/消息——这些消息较为罕见，不会影响跟踪压缩。

Trace Encoder

跟踪编码器（简称 TE）- 一种硬件模块，接收来自 hart 的执行信息，并生成跟踪消息/数据包流。

通过监控入口端口，追踪编码器可以确定程序流中断的发生，并判断该中断是否可推断。可推断的中断是指追踪解码器能够静态确定目标的中断，例如直接分支指令，其目标或偏移量包含在指令码中。不可推断的中断包括所有其他类型，如中断、异常和间接跳转指令。

Trace Sink

追踪接收器 - 硬件模块，用于接收追踪消息/数据包流，并将其记录到内存中或以某种格式转发。

  • SRAM Sink: Trace Encoder 将跟踪消息打包到固定宽度的跟踪字（通常为字节）中。这些跟踪字随后以循环缓冲区的方式存储在专用 RAM 中，该 RAM 通常位于片上。当 RAM 填满时，可以选择停止或回绕并覆盖先前的跟踪数据。
  • System Memory Sink: 跟踪编码器将跟踪消息打包成固定宽度的跟踪字（通常为字节）。随后，这些数据通过 DMA 型总线控制器以环形缓冲区的方式存储在为跟踪预留的系统内存范围内。当内存范围被填满时，可选择停止或循环覆盖先前的跟踪数据。这种类型的接收器也可用于通过例如 PCIe 或 USB 端口将跟踪数据传输到芯片外。
  • PIB Sink: 引脚接口块（Pin Interface Block），一种并行或串行的芯片外追踪端口，用于向追踪探针提供信号。跟踪编码器将跟踪消息发送至 PIB 接收器。每条消息通过特定协议以字节序列的形式片外传输。
  • ATB Bridge: ATB 桥接器允许将 RISC-V 跟踪数据发送至 Arm CoreSight 基础设施（而非本文件中定义的符合 RISC-V 规范的接收端），作为 ATB 发起方。对于仅使用 RISC-V 系统的环境，并不需要 ATB 桥接器。
  • Trace Funnel: 跟踪编码器可能将跟踪消息发送至跟踪集线器。集线器汇集来自每个输入（无论是 RISC-V 跟踪编码器还是另一个跟踪集线器）的跟踪数据，并将合并后的跟踪流发送至其指定的跟踪接收器或 ATB 桥接器，即上述一种或多种接收器类型。

Trace Funnel

追踪汇流模块(追踪排, 集线器) - 硬件模块，用于将来自多个追踪源（追踪编码器和/或其他追踪汇流模块）的追踪数据流合并为单一的追踪消息/数据包输出流。

跟踪控制

每个 Trace 组件（ENCODER，SRAM Sink, PIB Sink, ATB Bridge, Trace Funnel）由一组 32 位寄存器控制，这些寄存器占用最多 4KB 的地址空间。每个 Trace 组件的基地址必须对齐在 4KB 边界上。

应用

Trace32


Trace 是一种非侵入式、记录式的执行行为捕获机制。通过硬件模块实时记录处理器的指令流、数据访问或事件流，形成时序完整的执行日志（trace stream），供事后分析使用。从应用的角度看，trace 已经不仅仅供系统调试，更能作为应用使用。

  • 系统调试与故障诊断: 分析难以复现的死锁、竞态或崩溃；
    • 在复杂嵌入式系统或多核 SoC 中，软件崩溃或死锁难以复现，硬件追踪可提供完整的执行历史，帮助定位问题根源。
    • 支持“事后分析”（post-mortem debugging），即使系统已复位，只要 trace buffer 未被覆盖，仍可提取关键信息。
  • 性能剖析（profiling）与热点识别；
    • 精确统计函数调用次数、分支跳转行为、缓存命中率等，用于识别性能瓶颈。
    • 支持对实时系统（RTOS）的任务切换、中断响应时间进行测量，验证实时性约束。
  • 功能安全（ISO 26262）中的执行路径审计；
    • 在汽车电子（ISO 26262）、工业控制（IEC 61508）等领域，硬件追踪可提供可审计的执行证据，满足 ASIL 或 SIL 等级的验证要求。
  • 安全监控
    • 监控异常控制流转移（如缓冲区溢出、ROP 攻击），用于运行时完整性检测。
    • 在可信执行环境（TEE）中，验证关键代码是否按预期路径执行。

与DEBUG的区别：

维度 Debug（调试） Trace（追踪）
执行干预 主动干预：暂停 CPU、修改状态 被动记录：不中断正常执行
实时性 实时交互，依赖调试通信（如 JTAG） 实时记录，但分析通常为事后（post-mortem）
开销 高：暂停 CPU、上下文切换、通信延迟 低（理想情况下）：专用硬件流水线记录
信息粒度 点状快照：断点处的状态（寄存器、内存） 连续流：完整的执行路径（PC 流、分支、数据访问）
复现依赖 依赖可复现的错误场景 可记录不可复现的偶发故障
硬件依赖 需 Debug Module（RISC-V Debug Spec 定义） 需 Trace Unit（N/E-Trace）
带宽需求 低（仅状态查询） 高（需持续输出指令/数据流）
